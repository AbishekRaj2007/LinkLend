# SakshamScore

Alternate-data credit scoring for thin-file MSMEs: turns a business's GST filings,
bank transactions, EPFO payroll and existing obligations into an explainable
0‚Äì100 creditworthiness score ‚Äî no bureau history or collateral required.

## Run & Operate

- `pnpm install` ‚Äî install all workspace dependencies
- `pnpm run dev` ‚Äî run the frontend (port 4174) and API server (port 5000) together
- `pnpm run dev:web` ‚Äî run only the frontend (`sakshamscore`)
- `pnpm run dev:api` ‚Äî run only the API server
- `pnpm run typecheck` ‚Äî full typecheck across all packages
- `pnpm run build` ‚Äî typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` ‚Äî regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/db run push` ‚Äî push DB schema changes (dev only)
- `pnpm --filter @workspace/api-server run train` ‚Äî fit the pillar models on the
  deterministic synthetic dataset and regenerate the coefficient artifact (no DB needed)
- `pnpm --filter @workspace/api-server run validate` ‚Äî report AUC / Brier / ECE /
  accuracy per pillar and overall (no DB needed)
- `pnpm --filter @workspace/api-server run seed` ‚Äî seed Postgres with the synthetic
  dataset (needs `DATABASE_URL` + a pushed schema)
- Required env: `DATABASE_URL` ‚Äî Postgres connection string (only for the live API +
  seed script). Optional: `GROQ_API_KEY` ‚Äî enables LLM credit-memo narration; without
  it, a deterministic template memo is used.

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- API: Express 5
- DB: PostgreSQL + Drizzle ORM
- Validation: Zod (`zod/v4`), `drizzle-zod`
- API codegen: Orval (from OpenAPI spec)
- Build: esbuild (CJS bundle)

## Where things live

- **DB schema (source of truth):** `lib/db/src/schema/index.ts` ‚Äî the five raw tables
  (`msme_master`, `gst_returns`, `transactions`, `epfo`, `obligations`).
- **Scoring engine** (pure TypeScript, DB-independent): `artifacts/api-server/src/`
  - `synthetic/generate.ts` ‚Äî deterministic synthetic dataset (latent-health driven).
  - `features/` ‚Äî `computeFeatures` turns a bundle into the ~20-feature vector,
    completeness vector, and cross-source consistency ratios. `features/catalog.ts` is
    the single source of truth for feature names, pillar membership, weights, and
    neutral defaults.
  - `ml/` ‚Äî seeded RNG, logistic regression + Platt scaling, and metrics (AUC/Brier/ECE).
  - `scoring/` ‚Äî pillar evaluation, blended score + rating band (`score.ts`),
    exact coefficient-based reason codes (`explain.ts`), confidence (`confidence.ts`),
    consistency/fraud flag (`consistency.ts`), cashflow forecast + sustainable EMI.
    `scoring/model-artifact.generated.ts` holds the trained coefficients.
  - `data/store.ts` ‚Äî the only scoring path that touches Postgres.
  - `scripts/` ‚Äî `train.ts`, `validate-scores.ts`, `generate-data.ts`.
  - `lib/groq.ts` ‚Äî AI narration, quarantined to explanation only.
- **API routes:** `artifacts/api-server/src/routes/score.ts`
  (`GET /api/msmes`, `/api/msmes/:id/scorecard`, `/api/msmes/:id/memo`).

## Architecture decisions

- **Scoring core is pure and DB-independent.** Features/scoring operate on plain
  typed records, so the same code serves the live API (rows from Postgres) and the
  training/validation scripts (rows generated in-memory), and is fully unit-testable.
- **Training and validation never need a database.** The dataset is regenerated
  deterministically (seed 42); only the live API route and the seed script use Postgres.
- **Fixed weighted blend, not a learned meta-model.** The five pillar weights
  (25/30/20/15/10) are fixed for auditability. Pillars are individually Platt-calibrated;
  the blend is a transparent weighted sum ‚Äî treat `overallScore/100` as a risk *ranking*,
  not a literal probability.
- **AI is quarantined to narration.** `lib/groq.ts` only ever describes an
  already-computed scorecard; its system prompt forbids recomputing or re-ranking.
- **Synthetic validation only.** `outcome_label` is generated by the same latent
  process as the features, so metrics prove the pipeline recovers a planted signal,
  not real-world default prediction.

## Product

Given an MSME's raw data, SakshamScore returns a scorecard: an overall 0‚Äì100 score and
risk band, five pillar sub-scores (Business Vitality, Cashflow Health, Formalisation &
Compliance, Banking Behaviour, Obligations & Leverage), exact reason codes, a
data-completeness confidence level with an actionable next step, a cross-source
consistency/fraud alert, a suggested sustainable EMI from a 6-month cashflow forecast,
and an optional AI-narrated credit memo.
